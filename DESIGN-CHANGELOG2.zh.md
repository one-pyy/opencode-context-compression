# opencode-context-compression 设计改动项清单（第二轮补充）

> 这份文件记录 2026-04 本轮相对 `DESIGN.md` 既有内容新增的**运行时重放模型补充**。
>
> 它不是实现完成证明，也不是当前代码现状镜像；它的作用是把这轮对话里确认的新模型收敛成一份简洁的变更说明，供后续实现时快速比对。

---

## 1. 本轮补充的总原则

本轮的根本收敛点是：

1. **回到设计原型**：让模型选择“压缩什么 / 删除什么”，cache 只改变执行时机，不改变语义
2. **hook 主导语义**：当前有效 mark 集、覆盖关系、替换与 fallback 都由 hook 重放宿主历史得出
3. **数据库只存结果**：数据库的主职责是存放按 mark id 键控的 replacement 结果组，而不是提前持久化整套 mark/source 真相

---

## 2. mark 工具语义改动

### 2.1 `compression_mark` 退化为“返回 mark id 的占位工具”

- 工具调用后立即返回随机 mark id
- 一次调用只允许一个范围（可为单条消息）
- 如果需要多个标记，模型必须多次调用工具
- 当前目标模型下，不再要求调用时立即把 marks/source snapshots 全量写入 SQLite 作为 projection 真相源

### 2.2 `mode` 仍然是本次动作意图

- `mode=compact`
- `mode=delete`

delete 仍受当前 delete policy / allow-delete gate 限制；若当前不允许 delete，则本次 tool 调用在模拟执行阶段判为失败。

### 2.3 `allowDelete` 的职责收缩

- 不再把 `allowDelete` 当成 mark 的长期业务语义去传播
- 它只在 delete admission 阶段起作用
- 一旦 mark 被接受，后续重放与替换主要看 mark id 与其结果组

---

## 3. 当前有效 mark 的推导方式改动

### 3.1 从“读 mark 表”改为“重放历史 tool 调用”

每次 hook 都从头到尾重放历史里的 mark tool 调用，重新计算：

- 哪些 mark 合法
- 哪些 mark 被覆盖
- 哪些 mark 是错误调用
- 哪些 mark 当前顶层有效

### 3.2 覆盖规则明确为：大盖小，后盖前；相交报错

- 后 mark 包含或等于前 mark → 后者覆盖前者
- 仅相交 → 后者判错，tool 返回改写为错误消息
- 完全不相交 → 共存

### 3.3 “覆盖”只改变候选资格，不立即销毁旧结果

被覆盖的小 mark 仍然挂在大 mark 下方，用于后续 fallback。只有当更大 mark 自己已有结果时，小 mark 才真正不再显示。

---

## 4. 覆盖树与 fallback 改动

### 4.1 引入根节点

在所有合法 mark 之上新增一个无 mark id 的根节点，统一管理整棵覆盖树。

### 4.2 渲染算法收敛为统一递归规则

对任意节点：

- 自己有完整结果 → 直接使用自己，整棵子树不再展开
- 自己没结果 → 递归看子节点
- 子节点也没有 → 回原文

### 4.3 解决了“大 mark 未完成导致小结果闪回原文”的盲点

现在更大的 mark 尚未准备好时，可以继续显示下面较小但已完成的结果，而不是整段退回原文。

---

## 5. replacement 语义改动

### 5.1 compact 与 delete 共用同一 replacement 机制

- 两者都按 mark id 去数据库查结果组
- 两者都通过 hook 做替换
- 不同之处在于提示词与结果内容，不在替换算法

### 5.2 一个 mark 允许对应多条 replacement

当范围中含有不可压缩片段时，一次 compact 请求可以生成多条 replacement；但它们语义上仍是同一个 mark 的**完整结果组**。

### 5.3 结果组必须原子生效

- 要么整组都出现
- 要么整组都不出现
- 不允许在最终视图里出现半成品

---

## 6. “不可压缩”语义被重新精炼

本轮明确：

- “不可压缩”不是指该块永远不能进入更大压缩范围
- 它只表示该块的**内部内容不能被再次自由改写**

因此：

- compact 结果可以作为原子占位块，进入更大 compact 的输入
- delete 也可以整段删除它
- 但不能把它当成原始文本再次展开重写

---

## 7. 错误 tool 调用的语义改动

本轮明确区分两类情况：

1. **合法但尚无结果**
   - 仍在覆盖树中
   - 只是当前没有 replacement

2. **模拟执行阶段判错**
   - tool 返回改写为错误消息
   - mark id 不进入覆盖树
   - 不参与 token 统计
   - 不参与 replacement lookup

同时确认：错误消息仍是普通可见消息，后续可以被更大的合法 compact / delete 处理。

---

## 8. token 口径改动

### 8.1 新增“未压原始 token”口径

调度阈值现在更适合看：

- 当前有效 mark 覆盖范围中
- 还有多少原始内容尚未被已有结果接管

### 8.2 新增树上的传播规则

- 自己有结果 → 该节点整段原始 token 都视为已压，孩子不再继续向上传播
- 自己没结果但孩子有结果 → 子节点已压量求和
- 都没结果 → 已压量为 0

### 8.3 最终 prompt 负载仍单独统计

最终送给模型的真实 prompt token，不从树上反推，而是在最终视图完全渲染后整体再算一遍。

---

## 9. 当前模型下需要特别记住的例子

### 9.1 小结果先完成，大结果后覆盖但未完成

- 小 mark 有结果
- 大 mark 覆盖它，但还没结果
- 最终视图继续显示小结果，而不是回原文

### 9.2 等于范围的后 mark

- 后 mark 仍覆盖前 mark
- 但前 mark 可作为后 mark 的 fallback 子节点

### 9.3 只有交集的 mark

- 后 mark 判错
- 最终视图保留错误消息
- 不进入后续运算

### 9.4 compact 结果进入更大 compact

- 它作为原子占位块被引用
- 内部不可重写
- 最终可能使大 mark 生成多个 replacement 片段

### 9.5 delete 覆盖 compact 结果

- delete 不需要第二套替换机制
- 它只是另一种 replacement 结果类型

---

## 10. 对 DESIGN 主文的直接修订

本轮不是依赖“后章覆盖前章”的补丁式文档，而是要求 `DESIGN.md` 主文本身保持自洽。本轮已经把主文直接修到以下方向：

1. `allowDelete` 从长期持久语义位改回 delete admission gate
2. mark/source snapshot lookup 从 projection 主真相源改为历史重放 + mark id 结果组 lookup
3. “不能再次压缩”改精确为“不能再次内部改写，但可被更大范围包含，也可被 delete 覆盖”
4. 失败 mark 从“保留为可重试持久状态”改成区分“合法但无结果”和“模拟执行即报错”的两类语义
5. “无结果”不再粗糙描述为“回到原文”，而是精确定义为“当前节点不产生替换，保留原位置内容不变”
6. `compact` 对不可压缩片段的 XML + 占位符约束，以及缺失占位符后的 retry / fallback 流程，已直接写入主文

因此，`DESIGN.md` 的前文与第 15 章现在应当是相互一致、互相补充的关系，而不是冲突后再靠后章兜底。
