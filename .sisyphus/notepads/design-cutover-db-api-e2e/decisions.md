# 2026-04-06 runtime-config contract decisions
- Kept the config surface strictly aligned to `src/config/runtime-config.schema.json` plus DESIGN defaults; unsupported legacy fields such as `reminder.counter` are rejected instead of ignored for compatibility.
- Exposed small, reusable helpers from `src/config/runtime-config.ts` for downstream consumers: repo-root path resolution, compaction model-chain parsing, prompt asset loading, reminder prompt selection, and threshold readers.
- Treated compaction prompt and reminder prompts differently on purpose: compaction prompt may be a template asset, but reminder prompts must remain plain-text artifacts with no placeholder syntax.
- Added interface-level tests under `tests/e2e/interfaces/` to lock down precedence, schema/shape rejection, prompt selection, and fail-fast missing-asset behavior before later DB/interface/e2e tasks build on the loader.
- Chose a destructive schema-reset bootstrap for the sidecar DB: if the existing SQLite file contains legacy truth tables, runtime gate/job tables, incompatible non-table objects, or mismatched column layouts, bootstrap drops the user schema and recreates the four DESIGN-approved tables instead of attempting compatibility migrations.
- Kept restart recovery explicitly outside SQLite by resolving session DB paths in `state/`, session lock paths in `locks/`, seam/runtime logs as configured JSONL artifacts, and debug snapshots as optional `session_id.in.json` / `session_id.out.json` files under the configured snapshot directory.
- Treated `result_groups` + `result_fragments` as an all-or-nothing committed read model keyed by `mark_id`; failed compaction attempts do not persist failed rows, and a missing DB after deletion simply means replay starts from canonical history plus an empty result cache.
## 2026-04-06 hermetic e2e harness policy
- Default E2E policy is deny-by-default network access. Any test that tries live fetch, http, or https without an injected test transport must fail immediately with an unauthorized-network error.
- Session naming is normalized as e2e-<suite-slug>--<case-slug> and must also pass the repo path-safety single-segment rule before being used for evidence or future session-scoped state.
- Safe transport for repo-owned E2E is an injected scripted fixture with deterministic success, failure, and timeout outcomes plus call logging; this keeps future scheduler/runner tests hermetic until a real production transport exists.
- Evidence layout is standardized by a manifest that records convention version, session id, deny-by-default network policy, injected safe-transport policy, and Node test-runner ownership.
- For Task 1 specifically, replay rebuild is expressed as a boundary-level hydration operation from replay fixtures into the locked tables, not as a general-purpose repository contract; richer create/read/list/idempotent APIs are intentionally deferred to Task 2.

## 2026-04-06 Task 2 repository decisions
- Kept Task 2 inside `src/state/sidecar-store.ts` instead of introducing a second storage module, so the repository APIs and the locked schema bootstrap/rebuild helpers share one source of truth for DDL, payload hashing, and transaction handling.
- Exposed both `readResultGroup(markID)` and `getResultGroupByMarkID(markID)` on purpose to match the task wording without inventing broader query surfaces; both resolve to the same strict mark-id lookup semantics.
- Defined duplicate-write behavior as idempotent only when the stored committed group is byte-for-byte equivalent at the repository record level, including fragment ordering and derived payload hash; divergent rewrites for an existing committed `mark_id` are rejected.
- Chose fail-fast corruption reads for missing or out-of-order fragments because the plan locks same-mark complete reads; soft fallback would make partially visible result groups look valid and would violate the sidecar read-model contract.
- Refactored the state layer so `src/state/sidecar-store.ts` is only the public facade, with implementation organized under the `src/state/sidecar-store/` directory by concern (`schema`, `replay`, `repository`, `result-groups`, `visible-ids`, `helpers`, `types`). This keeps Task 1 + Task 2 behavior intact while giving later tasks smaller seams to extend.
- Reworked that split into a folder-based layout after user feedback: `src/state/sidecar-store.ts` stays as the stable import surface, but the implementation now lives under `src/state/sidecar-store/{schema,replay,repository,result-groups,visible-ids,helpers,types}.ts`. This keeps navigation natural without changing Task 1 + Task 2 behavior or forcing downstream callers onto deep import paths.

## 2026-04-06 Task 5 transport contract decisions
- Introduced a dedicated `src/compaction/transport/` directory instead of growing `src/compaction/runner.ts` into a monolith; `src/compaction/{input-builder,output-validation,runner}.ts` stay as thin facades while the transport seam is organized by concern.
- Kept the compaction runner contract deliberately narrow: one injected `CompactionTransport`, one attempt at a time, raw payload validation, and no model-chain fallback logic yet. Retry ordering stays a later task, but the typed `CompactionTransportRetryableError` surface is in place now.
- Chose runtime fail-fast over implicit defaults: `requireCompactionTransport()` throws a `CompactionTransportConfigurationError` that explicitly says injected safe transport is mandatory and a live executor is unavailable.
- Made malformed output a validation error instead of a transport error. The transport may return any raw payload, but only `validateCompactionTransportPayload()` can turn that into committed compaction content, which preserves the boundary between model execution and repo-owned output rules.
- Standardized recorded call outcomes as repo-owned primitives instead of leaking provider error objects: timeouts carry `timeoutMs`, retryables carry a stable `code?`, and abort/cancel are separated by origin (`caller` vs `transport`).

## 2026-04-06 Task 6 external contract decisions
- Implemented the Task 6 plugin entry as a thin adapter only: `src/index.ts` now loads runtime config and delegates all seam construction to `src/runtime/plugin-hooks.ts`, rather than embedding hook logic or tool admission directly in the entry file.
- Chose explicit external contract descriptor objects for each public seam (`messages.transform`, `chat.params`, `tool.execute.before`, `compression_mark`) so tests can lock call timing, visible side effects, error semantics, and replay/result-group/scheduler relationships without prematurely defining the full Task 7 internal architecture.
- Kept the default `compression_mark` admission conservative: `compact` is allowed, `delete` returns `DELETE_NOT_ALLOWED` unless a later runtime policy adapter is injected. This preserves the DESIGN admission boundary without inventing a new repo-owned config surface in Task 6.
- Made `messages.transform` own the in-place array mutation at the adapter layer even when the injected projector returns a fresh array. That keeps the host hook contract correct while still allowing later projection logic to live outside the seam adapter.

## 2026-04-06 Task 8 replay / visible-id decisions
- Moved visible-id formatting and stable suffix derivation into `src/identity/visible-sequence.ts` and made the sidecar helper reuse that logic. This keeps durable host allocations and projection-only IDs (`referable_*`, `reminder_*`) on one formatting contract instead of re-implementing the pattern in multiple layers.
- Kept `HistoryReplayReader` split between a generic `read(sessionId)` interface and a new repo-owned `createHistoryReplayReaderFromSources()/replayHistoryFromSources()` helper. That lets Task 8 implement real host/tool replay semantics without forcing projection tests or future runtime wiring to build `ReplayedHistory` by hand.
- Implemented mark legality in `PolicyEngine` instead of `ProjectionBuilder` so the builder can stay as an orchestrator: replay read -> policy classification -> durable visible-id allocation -> legal tree build -> result-group rendering -> reminder injection. This keeps the builder from collapsing into the monolith the user warned against.
- Treated reminder IDs as projection-only stable metadata derived from the anchor visible sequence, but kept reminder text itself prefix-free in the final message layer. That satisfies the Task 8 visible-id contract while preserving the DESIGN rule that reminder artifacts do not consume durable message-layer visible sequence slots.
- Deleted the old stripped `src/projection/messages-transform.ts` stub rather than carrying another dead entry point. The active seam remains `src/runtime/messages-transform.ts`, which now has a projection-backed projector adapter and envelope renderer for the real single-exit projection path.

## 2026-04-06 Task 9 compaction interface decisions
- Extended the compaction request transcript shape with source-sequence metadata and an optional opaque-slot marker so the compaction builder, validator, and runner share one frozen view of the range being executed. This keeps retry/fallback attempts deterministic and gives successful commits enough information to write projection-safe fragments.
- Defined recoverable runner failures narrowly: retryable transport errors, malformed payloads, and invalid compact output consume the same-model retry budget and can fall through to the next configured `compactionModels` entry; timeout, fatal transport errors, and abort/cancel stop immediately and leave state untouched.
- Kept `build.model` as the primary entry point for backward-compatible callers, but normalized the actual model chain as `[build.model, ...compactionModels]` with de-duplication so fallback ordering is explicit and stable in tests.
- Delete-mode validation intentionally bypasses compact-only placeholder retention checks. Once delete admission has already been allowed and the runner is executing `mode='delete'`, the replacement may cover opaque blocks wholesale and should commit as a single delete-style fragment over the frozen source range.

## 2026-04-06 Task 10 runtime decisions
- Kept `tool.execute.before` intentionally simple for Task 10: it performs only DCP-vs-passthrough lane selection (`compression_mark` -> `dcp`, everything else -> `passthrough`) and does not take over ordinary-chat waiting or projection work. The real observable wait semantics live in `SendEntryGate` on top of file-lock state.
- Wired the default plugin entry to a real history-backed scheduler using `input.client.session.messages(...)` plus the runtime lock directory derived from the plugin directory. This keeps `src/index.ts` as thin wiring while making `chat.params` default behavior real instead of static.
- Enriched `ChatParamsSchedulingMetadata` with `activeCompactionLock`, `pendingMarkCount`, and optional `dispatchedBatch` because those fields are the minimum runtime metadata needed to prove narrow scheduling and dispatch freeze behavior in tests without turning metadata into a cross-turn truth source.

## 2026-04-06 Task 11 recovery decisions
- Kept the production change surface minimal by fixing delete-style projection at the rendering seam only: committed `mode='delete'` result groups now render their stored delete notice text without a referable visible-id prefix, while `mode='compact'` continues to render referable replacement blocks. This matches DESIGN §10.2 / §15.17 without pre-solving Task 12’s broader success-path wiring.
- Recovery correctness remains anchored on Task 9’s rule: no committed result-group rows exist until transport output validates successfully. Task 11 therefore proves timeout, malformed payload, and retry-after-failure behavior through the existing atomic repository contract instead of introducing failed rows, watermark bookkeeping, or runtime-state tables.
