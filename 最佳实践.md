# AI Agent 项目执行控制最佳实践

> 本文档解决一个具体问题：你已经写了详细的设计文档，但 AI agent 在实现时仍然漂移。
>
> 核心论点：**漂移不是因为信息不够，而是因为执行控制缺失。** 信息和控制是两件不同的事，不能互相替代。

---

## 第一部分：为什么"说清楚了"还是会漂

### 1.1 Agent 的认知模型与人类工程师不同

人类工程师读完一份 200 页的设计文档后，会形成一个持久的心智模型。之后每次写代码，他调用的是这个心智模型，而不是每次都重新读文档。

Agent 没有这个能力。每次任务开始，它都在有限的上下文窗口里重新解读你的文档。这意味着：

- 文档越长，注意力越分散。LLM 对文档开头和结尾的注意力最强，中间最弱。
- 文档里的"例子"和"规范"对 agent 来说权重相近。你觉得例子只是辅助说明，agent 可能把它当成规范来执行。
- 文档里的内部张力（前文说 A，后文说 B）对人类来说可以靠上下文判断，对 agent 来说是一个分叉点——它会选一条路走下去，而你不知道它选了哪条。

### 1.2 漂移的三种发生机制

**机制一：范围蔓延**

你让 agent 改文件 A，它发现 A 依赖 B，于是顺手改了 B。B 的改动又触发了 C 的不一致，于是它又改了 C。最终你发现它改了 7 个文件，其中 5 个不在你的预期范围内。

原因：agent 的默认行为是"让代码一致"，而不是"只做你说的"。

**机制二：歧义补全**

你的设计文档里有两段话对同一个概念的描述不完全一致。agent 不会停下来问你，它会选一个它认为更合理的解释，然后按这个解释实现。

原因：agent 的默认行为是"合理推断然后继续"，而不是"遇到不确定就停下来"。

**机制三：层级混淆**

你的设计文档同时包含"永久不变的原则"和"当前版本的实现细节"。agent 无法区分这两者的权重，可能把一个临时的实现建议当成硬性约束来执行，或者把一个核心 invariant 当成可以灵活处理的建议。

原因：agent 没有"这条规则比那条规则更重要"的元认知，除非你显式告诉它。

### 1.3 更多信息不能解决的问题

以下问题无法通过"把设计文档写得更长"来解决：

| 问题 | 为什么更多信息无效 |
|---|---|
| Agent 在歧义点的选择行为 | 需要的是"遇到歧义时停下来"的行为规则，不是更多背景 |
| Agent 的执行范围蔓延 | 需要的是显式的 scope 边界，不是更详细的设计 |
| Agent 把目标设计当成当前实现 | 需要的是文档分层，不是在同一份文档里加更多标注 |
| Agent 在长文档里注意力不均匀 | 需要的是更短的执行输入，不是更长的参考文档 |

## 第二部分：文档分层——不同文档服务不同决策

### 2.1 核心原则

**一份文档只回答一类问题。** 如果一份文档同时回答以下四类问题，它就已经过载了：

1. 系统为什么这么设计？→ 设计文档的职责
2. 当前实现到了哪里？→ 状态追踪的职责
3. 下一步具体改哪些文件？→ 执行计划的职责
4. 这次任务怎么验收？→ 验收标准的职责

### 2.2 推荐的四层文档结构

#### 第一层：设计契约（DESIGN.md）

只放"几年后看仍然应该成立"的内容。

应该包含：
- 系统的核心 invariant（例如"宿主历史是唯一真相源"）
- 关键概念的权威定义（例如 `allowDelete` 的语义）
- 模块间的接口契约
- 方案权衡的记录（为什么选 A 不选 B）
- 顶层验收标准

不应该包含：
- 当前的文件名、配置字段名、prompt 文件名。原因：这些会随实现演化，放在设计文档里会导致文档频繁修改，每次修改都可能引入不一致。
- 模块内部的实现建议。原因：这是实现者的自由度，写进设计文档会让 agent 把建议当成约束。
- 迁移步骤和过渡说明。原因：迁移是一次性的，设计文档是长期的，混在一起会让读者分不清哪些内容是永久有效的。
- 测试文件名和测试用例细节。原因：测试会随实现变化，放在设计文档里会导致文档和代码不同步。

#### 第二层：设计变更记录（DESIGN-DELTA-YYYY-MM-DD.md）

只记录设计层面的变更，不记录实现细节。

每条变更必须包含：
- 改了什么（旧语义 → 新语义）
- 为什么改
- 影响哪些设计契约
- 是否已经过 review

这份文档不是给 agent 执行的，是给 owner / reviewer 看"设计发生了什么变化"。

#### 第三层：执行计划（IMPLEMENTATION-PLAN.md）

只管"怎么分阶段做完"。

应该包含：
- Phase 划分（每个 phase 的目标和边界）
- 每个 phase 涉及的模块
- Phase 之间的依赖关系
- 风险和前置条件

不应该包含：
- 设计原理的解释。原因：执行计划的读者已经读过设计文档，重复解释会增加长度、稀释注意力，而且容易和设计文档产生措辞不一致。
- 具体的代码改动细节。原因：这是 task brief 的职责。

#### 第四层：任务简报（TASK-BRIEF）

**这是 agent 真正拿来干活的文档。** 每次只给 agent 一个小而硬的 brief。

模板（后文第三部分详细展开）：

```markdown
## Goal
一句话说明本次任务的唯一目标。

## In Scope
列出本次允许修改的文件和模块。

## Out of Scope
列出本次禁止触碰的文件和模块，附理由。

## Invariants
列出本次必须保持不变的系统约束。

## Acceptance Criteria
列出可机器验证的通过条件。

## Escalation Rule
遇到以下情况时，停止执行并报告：[具体列出]
```

### 2.3 四层之间的关系

```
设计契约（稳定，很少改）
    ↓ 引用
设计变更记录（每次设计变更时写）
    ↓ 驱动
执行计划（每个项目周期写一次，可以调整）
    ↓ 拆解为
任务简报（每个具体任务写一份，用完即弃）
```

关键规则：**信息只向下流动，不向上回写。** 如果实现过程中发现设计需要修改，不是直接改设计文档，而是先写一条 design delta，经过 review 后再更新设计契约。

## 第三部分：任务简报——给 Agent 的执行控制核心

### 3.1 为什么任务简报是最关键的一层

设计文档解决的是"说清楚"的问题。任务简报解决的是"控制执行"的问题。

Agent 漂移的直接原因几乎都可以归结为：任务简报缺失或不够硬。具体来说：

- 没有明确的 scope 边界 → 范围蔓延
- 没有 escalation rule → 歧义补全
- 没有 invariant 约束 → 破坏已有契约
- 没有可验证的 acceptance criteria → 无法判断是否完成

### 3.2 任务简报模板（完整版）

```markdown
# Task Brief: [简短标题]

## Goal
[一句话。不是"实现 reminder 系统"，而是"把 reminder-service.ts 的 token 计数从 counter.source 改成 softRepeatEveryTokens / hardRepeatEveryTokens"。]

## Design Reference
[引用设计文档的具体章节，例如 "DESIGN.md §3.4"。不要复制设计文档的内容到这里。]

## In Scope
- `src/projection/reminder-service.ts`：修改 token 计数逻辑
- `tests/projection/reminder-service.test.ts`：更新对应测试

## Out of Scope（禁止触碰）
- `src/projection/projection-builder.ts`。原因：本次只改 reminder 的 token 计数，不改 projection 渲染逻辑。
- `src/config/runtime-config.ts`。原因：配置面的迁移是另一个独立任务。
- 任何 prompt 文件。原因：prompt 文件的拆分是另一个独立任务。

## Invariants（本次必须保持不变）
1. 所有现有测试必须继续通过（除了本次明确要修改的测试）。
2. `system` 和短 `user` 消息不计入 reminder token。
3. `tool` 消息计入 reminder token。

## Acceptance Criteria
1. `reminder-service.test.ts` 中的 token cadence 测试全部通过。
2. `npm run test` 全量通过，无回归。
3. `npm run build` 成功，无类型错误。

## Escalation Rule
遇到以下任何情况时，**停止执行并报告**，不要自己决定：
- 发现需要修改 Out of Scope 中列出的任何文件。
- 发现 DESIGN.md 中的描述与当前代码不一致。
- 发现本次改动会破坏 Invariants 中列出的任何约束。
- 对 `allowDelete` 的语义有任何不确定。
```

### 3.3 好的任务简报 vs 坏的任务简报

**坏的任务简报（会导致漂移）：**

```markdown
参考 DESIGN.md 第 3、5、8、15 章，实现 reminder 系统的 token cadence 改造，
同时处理 allowDelete 语义收敛和 prompt 文件拆分。
确保所有测试通过。
```

这个简报的问题：
- 引用了 4 个章节，agent 需要自己判断哪些内容适用于本次任务
- 同时包含 3 个独立的改动（token cadence、allowDelete、prompt 文件），scope 太大
- "确保所有测试通过"是意图，不是可验证的指令
- 没有 out of scope，agent 会自由扩展改动范围
- 没有 escalation rule，agent 遇到歧义会自己推断

**好的任务简报（控制漂移）：**

```markdown
## Goal
把 reminder-service.ts 的 token 计数从 counter.source 改成
softRepeatEveryTokens / hardRepeatEveryTokens。

## In Scope
- src/projection/reminder-service.ts
- tests/projection/reminder-service.test.ts

## Out of Scope
- allowDelete 相关逻辑（另一个任务）
- prompt 文件拆分（另一个任务）
- projection-builder.ts（本次不碰）

## Escalation Rule
如果发现 token 计数改动需要动 Out of Scope 中的任何文件，停下来告诉我。
```

两者的信息量差距很大，但第二个的执行边界硬得多。

### 3.4 任务简报的粒度原则

一个任务简报应该满足以下全部条件：

- **单一目标**：只做一件事。如果你需要用"同时"或"并且"来描述目标，说明应该拆成两个任务。
- **可验证完成**：有明确的、可机器执行的验收条件（测试通过、构建成功、lint 通过）。
- **边界封闭**：In Scope 和 Out of Scope 合在一起，覆盖了 agent 可能触碰的所有文件。没有灰色地带。
- **歧义有出口**：Escalation Rule 覆盖了你能预见的所有歧义点。agent 遇到不确定时的行为是"停下来问"，而不是"合理推断"。

## 第四部分：执行流程——从设计到交付的阶段门禁

### 4.1 六阶段流程

```
阶段 1：立项        → 产出：Problem Statement
阶段 2：设计        → 产出：Design Doc（设计契约）
阶段 3：设计评审    → 产出：Approved Design + Design Delta（如有修改）
阶段 4：拆执行      → 产出：Implementation Plan + Task Briefs
阶段 5：逐任务执行  → 产出：代码变更 + 验收结果
阶段 6：阶段性合并  → 产出：更新后的状态追踪
```

每个阶段之间有一个门禁。门禁的作用是：**确保上一阶段的产出质量足够，再进入下一阶段。** 不是走形式，而是防止"带着模糊的设计开始写代码"。

### 4.2 各阶段的门禁条件

| 阶段转换 | 门禁条件 |
|---|---|
| 立项 → 设计 | 问题定义清晰，成功标准可验证，非目标已列出 |
| 设计 → 设计评审 | 设计文档覆盖所有核心概念，无内部矛盾，invariant 已显式列出 |
| 设计评审 → 拆执行 | 设计已被 review 并批准，所有 open question 已关闭或显式标记为 deferred |
| 拆执行 → 逐任务执行 | 每个任务有独立的 task brief，brief 满足 §3.4 的粒度原则 |
| 逐任务执行 → 阶段性合并 | 所有 acceptance criteria 通过，无回归，无未报告的 design delta |

### 4.3 执行阶段的具体操作

对于每个 task brief：

**步骤 1：执行前确认**
- Agent 读取 task brief
- Agent 确认理解 goal、scope、invariants
- 如果有任何不清楚的地方，在这一步提出，不要开始写代码

**步骤 2：执行**
- 只修改 In Scope 中列出的文件
- 遇到 Escalation Rule 中的情况时停止
- 不做任何 Out of Scope 中的改动

**步骤 3：自验证**
- 运行 Acceptance Criteria 中列出的所有检查
- 报告结果，不美化、不省略

**步骤 4：Review against design**
- 不是问"跑通了吗"
- 是问"是否违反了 Invariants 中列出的约束"
- 是问"是否引入了设计文档中没有定义的新语义"

**步骤 5：处理偏差**
- 如果实现过程中发现设计需要修改：不直接改设计文档，写一条 design delta
- 如果实现过程中发现 scope 需要扩大：不直接扩大，回到 task brief 重新拆分

## 第五部分：防御性任务设计——预防 Agent 的典型失败模式

### 5.1 已知失败模式与对策

| 失败模式 | 表现 | 对策 |
|---|---|---|
| 范围蔓延 | 你让它改 1 个文件，它改了 7 个 | 在 Out of Scope 中显式列出禁止触碰的文件，附理由 |
| 歧义补全 | 设计文档有两种解读，它选了错的那个 | 在 Escalation Rule 中列出已知歧义点，要求遇到时停止 |
| 层级混淆 | 把"目标设计"当成"当前实现"来读 | 文档分层，task brief 中只引用设计文档的具体章节，不复制内容 |
| 过度重构 | 你让它修 bug，它重构了整个模块 | 在 task brief 中写明"只做最小改动，不重构" |
| 沉默偏差 | 它偏离了设计但没有报告 | 在 Escalation Rule 中要求"发现设计与代码不一致时必须报告" |
| 测试删除 | 测试不过，它删了测试 | 在 Invariants 中写明"不得删除或跳过现有测试" |
| 类型压制 | 类型报错，它加了 `as any` | 在 Invariants 中写明"不得使用 as any / @ts-ignore / @ts-expect-error" |

### 5.2 Escalation Rule 的写法原则

Escalation Rule 是防止漂移的最后一道防线。写法原则：

**必须具体。** 不要写"遇到不确定的情况时停下来"——agent 不知道什么算"不确定"。要写"遇到以下具体情况时停下来"，然后逐条列出。

**必须覆盖已知歧义。** 如果你的设计文档里有已知的内部张力（例如前文和后文对同一概念的描述不完全一致），在 escalation rule 里显式列出这些概念。

**必须包含兜底条款。** 在具体条目之后，加一条："如果遇到本 brief 未覆盖的任何设计疑问，停止执行并报告。"

示例：

```markdown
## Escalation Rule
遇到以下任何情况时，停止执行并报告：
1. 发现需要修改 Out of Scope 中列出的任何文件。
2. 发现 DESIGN.md §3 中的 reminder 语义与当前代码不一致。
3. 对 `allowDelete` 是"admission gate"还是"持久语义位"有任何不确定。
4. 发现本次改动会导致现有测试失败（非本次预期修改的测试）。
5. 遇到本 brief 未覆盖的任何设计疑问。
```

### 5.3 不可接受的借口清单

在 task brief 中可以显式列出 agent 不被允许使用的借口。这看起来不寻常，但在实践中非常有效：

```markdown
## 不可接受的借口
- "根据我的理解，这样做是合理的" → 你的理解不是验证。引用设计文档的具体章节。
- "为了保持一致性，我顺便改了 X" → 一致性不是本次任务的目标。只做 In Scope 中的改动。
- "测试已经通过了" → 列出具体通过了哪些测试，附运行输出。
- "这个改动很小，应该没有影响" → "应该"不是验证。运行完整测试套件。
```

## 第六部分：完整示例——从设计到任务简报的拆解过程

### 6.1 场景

你有一个设计文档，其中定义了 reminder 系统需要从"按消息计数"迁移到"按 token 计数"。同时还需要把 2 个 prompt 文件拆成 4 个，以及把 `allowDelete` 的语义从"持久位"收敛为"admission gate"。

### 6.2 错误做法：一个大任务

```markdown
请按照 DESIGN.md 实现 reminder 系统的完整迁移，包括 token cadence、
prompt 文件拆分、allowDelete 语义收敛。
```

这会导致 agent 同时处理 3 个独立的改动，scope 巨大，任何一个子任务的歧义都会影响其他子任务。

### 6.3 正确做法：拆成独立任务

**执行计划：**

```markdown
## Phase 1：配置面迁移
- Task 1.1：在 runtime-config 中新增 softRepeatEveryTokens / hardRepeatEveryTokens 字段
- Task 1.2：在 runtime-config 中用 allowDelete 替换 route

## Phase 2：Reminder 核心逻辑
- Task 2.1：reminder-service 的 token 计数改用新字段
- Task 2.2：reminder prompt 从 2 文件拆成 4 文件

## Phase 3：allowDelete 语义收敛
- Task 3.1：mark tool 的 allowDelete 改为 admission gate
- Task 3.2：移除 mark/replacement 中的 allowDelete 持久字段

## Phase 4：验证
- Task 4.1：全量测试 + 构建 + design audit
```

**然后为每个 Task 写独立的 task brief。** 每个 brief 只引用设计文档的相关章节，只列出该任务涉及的文件，只定义该任务的验收标准。

### 6.4 Task 2.1 的完整 brief 示例

```markdown
# Task Brief: Reminder Token Cadence 迁移

## Goal
把 reminder-service.ts 的 token 计数从 counter.source / counter.*.repeatEvery
改成 softRepeatEveryTokens / hardRepeatEveryTokens。

## Design Reference
DESIGN.md §3.4（阈值与重复 cadence 规则）

## In Scope
- src/projection/reminder-service.ts：修改 token 计数和 cadence 判断逻辑
- tests/projection/reminder-service.test.ts：更新 token cadence 相关测试

## Out of Scope（禁止触碰）
- src/config/runtime-config.ts。原因：配置面迁移是 Task 1.1，本次假设新字段已存在。
- src/projection/projection-builder.ts。原因：projection 渲染不在本次范围。
- prompts/ 目录下的任何文件。原因：prompt 拆分是 Task 2.2。
- allowDelete 相关逻辑。原因：语义收敛是 Phase 3。

## Invariants
1. system 和短 user 消息不计入 reminder token（DESIGN.md §3.2）。
2. tool 消息计入 reminder token（DESIGN.md §3.2）。
3. 不得删除或跳过现有测试。
4. 不得使用 as any / @ts-ignore / @ts-expect-error。

## Acceptance Criteria
1. reminder-service.test.ts 中所有 token cadence 测试通过。
2. npm run test 全量通过。
3. npm run build 成功。

## Escalation Rule
遇到以下情况时停止执行并报告：
```

## 第七部分：自检清单

写完任务简报后，用以下清单检查：

1. **单一目标检查**：Goal 能否用一句话说完，且不包含"同时"或"并且"？
2. **Scope 封闭检查**：In Scope + Out of Scope 是否覆盖了 agent 可能触碰的所有文件？
3. **可验证检查**：Acceptance Criteria 中的每一条是否都可以通过运行一个命令来验证？
4. **歧义出口检查**：Escalation Rule 是否覆盖了你能预见的所有歧义点？
5. **禁止事项检查**：Out of Scope 中的每一条是否都附了理由？
6. **层级分离检查**：Brief 中是否避免了复制设计文档的内容（只引用章节号）？
7. **Invariant 检查**：是否列出了本次必须保持不变的系统约束？
8. **最小化检查**：Brief 的总长度是否在 50 行以内？如果超过，考虑拆成两个任务。

## 附录：核心原则速查

| 原则 | 一句话 |
|---|---|
| 信息 ≠ 控制 | 说清楚了不等于控制住了，这是两件不同的事 |
| 文档分层 | 不同文档服务不同决策层，不要混在一起 |
| 任务简报是核心 | Agent 拿来干活的不是设计文档，是 task brief |
| 边界要硬 | Out of Scope 比 In Scope 更重要 |
| 歧义要有出口 | 遇到不确定时的正确行为是停下来，不是推断 |
| 禁止胜过建议 | "不要做 X"的遵从率远高于"尽量做 Y" |
| 偏差要显式 | 实现偏离设计时，写 design delta，不要偷偷继续 |
| 验收要可执行 | "确保质量"是意图，"npm run test 通过"是指令 |
